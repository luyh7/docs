前言

顾名思义，这里没有任何高深的知识，没有复杂的命令行，很多可能遇到的问题也不会有什么很好地解决方案，纯粹只是简单的介绍Git怎么从安装到使用。为了降低入门门槛，所以这篇教程将会以图形程序介绍为主，尽量避免任何的命令行操作。

特别提醒

尽管在这里的介绍中会以图形界面介绍为主，但是强烈建议使用命令行进行操作。不仅仅是因为命令行操作更快，更是因为各种GUI程序都隐藏了一些操作来简化使用，这会导致很多不可预知的结果产生。

另外，关于什么是版本控制、什么是Git等等内容不在这个系列文章的讨论范围，有兴趣的可以自行查找。

安装

准备工作

不管安装什么软件，必要的准备工作还是需要的。首先要把电脑准备好，当然啦这里不是说你有台电脑开机了就行，一般都是Windows系统，所以请先确保一些基本依赖已经装好了，比如.NetFramework4、VC2010等等（Win7一般这些都自带了）。

然后准备好各种安装包。本文主要说的是两个安装包，一个是Git，一个是SourceTree，如果嫌给的链接下载慢或者懒得去，可以上公司共享盘找最新的安装包。

为什么使用最新版本

人都是有惰性的，所以程序升级一般人是很不喜欢的，而且企业用户为了保证服务稳定，也很少会进行软件升级。但是对于一些工具类的软件，经常性的升级不单单可以修复现有的一些问题，还可以带来更丰富的功能体验，甚至大大提升现有的工作效率，很多时候都是利大于弊的，因此对于一些工具类的软件我推荐尽可能的升级到最新的稳定版。

为了尽可能的减少不必要的问题，请一定先装Git，再装SourceTree。同时在Git从1.x升级到2.x时，先卸载旧的版本。

安装图解

Git

安装过程没什么好说的，基本都是无脑下一步就好，但是有些关键的步骤还是需要截图说明一下的（不想看说明的请直接按图操作）。

这是一些基础选项界面，主要是一些快捷方式的生成选项。基本上按照上图的样子勾选就好了，因为不是很重要的选项，也就不一一说明了。

这个是用来选择命令行运行方式的。第一个选项表示Git命令只会在Git Bash程序里生效，不会修改你的环境变量，也就是说在CMD里面不能运行Git命令。第二个表示会把Git命令加入系统环境变量，但是只会添加git这一个命令。最后一个表示不但会把Git命令加入环境变量，还会添加额外的一些Unix命令。基本上为了稳妥起见，请选择第一个。

解决换行符问题的选项。第一个表示会根据不同的操作系统签出不同的换行符，第二个表示提交时会把换行符换成Linux格式，第三个表示不做任何处理。推荐选第二个，绝大多数的文本编辑器就算在Windows上也是可以正确处理Linux系统下的换行符的。

SourceTree安装十分简单粗暴，只有一个地方需要注意。

其他GUI程序

 除开SourceTree以外，还是有很多其他的优秀的GUI程序的，比如Git自带的gitk就不错，我刚接触Git的时候用的GitExtensions也是一个很好的GUI。不过上述的两个毕竟有些简陋，在免费的GUI里面SourceTree和SmartGit算是两个佼佼者了，这里选用SourceTree作为教程讲解，有历史原因，也是因为个人觉得SourceTree漂亮一点。

基础配置

ssh-key生成

首先在开始前，需要生成一个SSH key作为身份认证，这是全文少数需要使用命令行的地方之一。

打开Git Bash命令行，输入以下的命令，然后一路回车即可。特别提醒：邮箱记得用自己的真实邮箱

?
ssh-keygen -t rsa -C "yourname@aobi.com"

生成的Key存放在用户目录下的.ssh文件夹里，比如我的就放在这个目录里：C:\Users\jiangpei\.ssh。

打开Key存放的目录，如果简单的按照上述流程来操作，应该会看到至少两个文件——id_rsa和id_rsa.pub，现在你还不能够使用这对秘钥来拉取远程仓库，你还需要把id_rsa.pub的内容添加到远程服务器才行（比如gitlab或者github这类托管仓库），以此确保你可以正常的使用SSH。

可以参考gitlab的帮助文档，上面比较清晰的说明了如何在GitLab上面添加SSH公钥

Git的配置文件位于本机的用户目录下，比如C:\Users\jiangpei\.gitconfig，这个文件内容很简单，需要说的地方不多，直接贴出模板供诸位参考吧，切记不要无脑复制，切记不要无脑复制，切记不要无脑复制。

为什么是英文

作为一个对英文深恶痛绝的人，我其实是很不喜欢用英文界面的软件的。但这里还是推荐使用英文界面，因为Git很多操作都没有一个官方固定的中文翻译，很多翻译极其容易引发歧义，而使用英文界面就没有这个问题。

简单使用

任何的版本控制系统都有一些共同的功能，比如签出某个版本的文件、保存修改、查询修改历史等等。所以一切的使用手册都是围绕这些基本操作来的。

初始化

所谓初始化就是把一个Git项目给创建出来，要么是获取远程代码，要么是把本地目录改造成Git项目。因为我们多数都是把远程已有的项目拿到本地，所以教程就以此为主。

打开SourceTree，点击Clone/New按钮，在弹出的对话框中填选对应的远程仓库地址，然后点击Clone按钮，如下图所示

注意

如何获取远程仓库地址会在另外的文章里说明，现在就请不要过于纠结。

等待clone完成后就会看到如下的界面

目前对于这个界面没有什么好说的，初始化后点击红圈里面的按钮，就可以打开对应项目的文件夹，对里面的文件进行操作了。到这里初始化工作就完成了，可以暂时忘掉Git进行常规的文件编辑操作，直到一次修改完成。

提交修改好的文件

现在假设你已经把文件编辑好了，准备提交并分享给别人，或者你仅仅只是希望Git能帮你保存好现在的工作。这时候你就需要用到提交（Commit）功能。

回到SourceTree，因为你已经对Git项目的文件做了修改，所以现在SourceTree的界面应该是这个样子的

把注意力放到右边，总共有三块区域。最右边的是当前文件的修改内容，绿色表示新增的行，红色表示删除的行。跟提交相关的是中间部分，分为上下两栏，一个叫Staged files，一个叫Unstaged files。Staged files表示即将提交的文件，Unstaged files表示你做出了修改，但是还没准备要提交的文件。

暂时不要去关心为什么Git要区分准备提交的文件和还没准备提交的文件，这涉及到Git的设计原理，超出了本文范畴。

在Unstaged files里面选择你想提交的文件，当你所有需要提交的文件都出现在Staged files里面的时候，就可以点击左上的Commit按钮，进行提交了。

Commit Message，也就是提交信息是必须要写的，如果不知道怎么填写提交信息，请参考另一篇文章，教你如何在Commit时有话可说。填写完Commit Message之后，点击Commit按钮就完成了一次提交。

推送你的更改

Git有一个核心的概念，叫做everything is local。也就是时候你的绝大部分操作都是在本地完成的（这也就是为什么Git比SVN快速很多），所以上面的提交操作也是发生在本地的，因此别人并不能看到你所做的修改，你需要将你的更改push到远程仓库，才能与他人共享你的更改。

在SourceTree里push你的更改也是件很简单的事情，回到界面，当你已经commit一次修改之后，界面应该是下面的样子，可以很明显的看到push按钮上有了一个数字，表示你有多少个commit没有push到远程。

点击push按钮，就会弹出推送窗口，这个窗口要说的不多，对于初学者来说这个对话框的信息没有什么值得注意的，直接点击OK就好。

多说几句

这个对话框的信息其实是很丰富的，比如repository、origin、branch、tags、track等等名词都是git里面的重要概念，尤其是branch（分支），这里为了简化操作，所以刻意忽略了这些重要概念。

特别注意

很多时候在push代码的时候都会遇到本地版本太旧的问题，这个时候都需要先同步远程的更改才能继续下去。怎么同步远程的请看下面的内容。

同步远程的更改

上述的流程已经成功的把你的更改分享给别人了，接下来说一下怎么获取别人的更改。

首先，要确保自己的工作已经提交了（commit），否则在获取别人更改的时候会报错。然后点击pull按钮，弹出的对话框通常也是不需要理睬的，直接OK就好。

冲突的解决

一般来说，如果一个项目只有你一个人更改，几乎是不会发生冲突的，但是多人共同编辑就很难保证了，所以解决冲突是十分必须的技能。

当pull代码出现冲突时，大致的界面会是这个样子，注意带有感叹号的文件，那就是出现了冲突的文件。

解决冲突最简单的方式某过于取消某一边的修改，只选择某一个人的修改。在上面的例子里，要实现这个操作是很容易的，只需要右键冲突的文件，选择Resolve Conflicts–>Resolve Using 'Mine/Theirs'，就可以选择使用我的版本还是别人的版本。选定版本后，冲突是解决了，但是还是需要手动点一下commit提交的（此处有默认填好的提交信息，一般是不用修改的啦），之后的流程就和前面说的一样了。

如何区分mine和theirs

其实这是个很难回答的问题，早期我用这种方式解决冲突的时候也常常遇到这个问题，只能两个都试试，看哪个是对的就用那个。后来使用多了，也就渐渐知道了些规律，按照目前的使用流程，多半是以合并产生的冲突居多，这个时候你所处的分支必然就是mine，或者你在pull更改时产生了冲突，那么毫无疑问你本地就是mine。

除开上面这种简单粗暴的方式以外，你也可以手动编辑冲突的文件，把文件内容调整成自己想要的模样来达到解决冲突的目的。其余的流程也和上面的类似，就不多说了。

进阶

接入GitLab

公司使用GitLab作为Git中央仓库，其实接入和普通的远程仓库没有太多不同，会单独成文有历史原因，另外跟证书相关的配置也不好在这里说明，请移步让Git接入公司的GitLab仓库一文了解。

一些有用的教程链接

《Pro Git》中文版：最为完整详细的Git教程，新手推荐阅读前三章即可，多了也没啥用处。

廖雪峰版教程：号称史上最易懂的教程，虽然时间有些久远，但是不能掩盖其光辉，新手推荐。

GitHub入门教程：15分钟的入门级教程，交互式操作，边用边学，适合刚刚接触Git命令行的人。

从0开始学习GitHub:：以GitHub为基础来介绍Git的入门级教程，内容较新，适合啥都不懂的小白。

GitLab帮助文档：GitLab自带的帮助文档，对于如何更好的使用GitLab有很详细的描述，虽然是全英文的。闲暇时间有做一些文档的翻译，https://gitlab.bt/jiangpei/gitlab-doc，欢迎感兴趣的或者无聊的人士参与协助。

猴子都能懂的Git入门：号称连猴子都可以教会的Git入门文档，个人觉得不错，推荐之。
