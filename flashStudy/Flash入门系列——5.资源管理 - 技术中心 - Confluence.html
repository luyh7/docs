Flash入门系列——5.资源管理

转至元数据结尾
  • 由 未知用户 (yangjunxiong)创建, 最后修改于三月 25, 2014
转至元数据起始

好的Flash网页游戏必然有好的用户体验，而好的用户体验体现在程序上来说就是程序运行流畅。那么，怎样的才算是流畅呢？

流畅就是用户感觉不到卡、顿等不舒服的现象。一般来说，卡有两种情况：一种是客户端卡，这种现象主要是CPU或内存不够用造成的；另一种是网络卡，这个一般是由于网络延迟造成的，它对即时战斗类游戏的影响特别明显。而网络延迟是网络游戏中一个非常大难题，本文作为入门文档，就不讨论了。本文主要讨论前者——由于CPU或内存影响的游戏体验问题，即怎样管理我们的CPU和内存，让我们的游戏更加流畅。最后还有一个话题就是文件，文件这种资源对于我们Web Game来说也是极其重要的。

对此，我将本文讨论的内容分为以下三种：

  • 内存：分配、回收和泄露
  • CPU：CPU资源不够用的原因和怎样解决
  • 文件：文件大小的控制，加密

一、内存

大家都知道，当系统内存不够用的时候，系统会出现卡、顿等现象。另外，当Flash Player占用的内存很大时，也会消耗比较大的CPU资源去管理自己的内存，这也是我们不想看到的。所以，合理管理我们Flash程序的内存是非常重要的，

内存管理一般分为两个部分：分配和回收。

1、分配

至于Flash是怎样分配内存的，这部分内容可以在网上查到，并且在编程中很难用到，所以这里就不讨论了，有兴趣的同学可以在后面的附件中看到相关内容。但是分配内存资源时有一个大的原则需要注意，那就是节约。

节约的方式有很多种，下面主要提出三类：

  • 节约内存：选择适当的显示对象和原始类型
     

    ?
    trace(getSize(new Shape()));
    // output: 248
    trace(getSize(new Sprite()));
    // output: 424
    trace(getSize(new MovieClip()));
    // output: 456
     
    var a:Number;
    trace(getSize(a));
    // output: 8
    var b:int;
    trace(getSize(b));
    // output: 4
    var c:uint;
    trace(getSize(c));
    // output: 4
    var d:Boolean;
    trace(getSize(d));
    // output: 4
  • 对象重用：对象池、变量的申明放到循环外、一个BitmapData多个Bitmap共用等等重用的方式。而且对象重用会降低内存申请的次数，间接降低的垃圾回收的次数。而垃圾回收是比较耗CPU资源的，所以对象重用也是减少CPU卡的一种方式。
  • 事件与回调：事件机制会创建大量的对象，并且事件也没有回调效率高，这个原则在我们游戏里要求不高，但要了解。

以上三点，对象重用是重点。而对象重用的技术里面，对象池技术是相对比较陌生的，但在游戏中却是很常用的。下面将详细介绍一下。

对象池：简单来说就是将用过的对象保存起来，不立即销毁，当再次需要使用同一种类型的对象时，不另外创建，而是从原来保存的对象库里面取出来重复使用。这种技术在需要大量创建和销毁某一种类型的对象的编程模型里面经常用到。简单的实现过程如下：

?
public class ObjectXPool
{
    private var _objectList:Array = [];
    public function ObjectXPool()
    {
    }
     
    public function getObject():ObjectX{
        if(_objectList.length == 0){
            return new ObjectX();
        }
        return _objectList.pop();
    }
     
    public function returnObject(obj:ObjectX):void{
        obj.reset();
        _objectList.push(obj);
    }
}

一般用Array或Vector保存对象，如果池子里面没有对象，就生成一个返回。归还对象时，记得重置对象。这里只是简单的实现，对象池有很多种写法，大家可以灵活运用。

2、回收

简单来说，Flash会将程序不需要的内存回收起来，还给系统。这里需要明确的是，回收只针对垃圾，没人要的内存资源。什么资源才是垃圾资源呢？对于这个问题，本文将花大部分篇幅去解释。要解释垃圾，咱们得先看看什么叫做引用。

（1）引用

引用可以理解为C/C++中的指针，可以认为他保存了一个地址。除了基本数据类型（int、Number、Boolean...）使用值传递以外，所有对象都是都是通过引用传递。

?
public function test()
{
    var sp:Sprite = new Sprite();
}

如上面的代码：首先，new Sprite()从堆里面申请了一坨内存，把Sprite对象放进去。var sp:Sprite从栈中声明了一个变量，该变量保存了刚才那坨内存的首地址，也就是引用。

?
var sp2:Sprite = sp;

这句代码的意思是将sp的地址复制给sp2，这里记住是地址的复制，而不是对象Sprite。所以sp2和sp表示的是同一个对象。

?
var sp:int = 4
var sp2:int = sp;

而同样的代码，如果将Sprite换成int，就是值传递了。sp2和sp的值都是4，并且他们的4是存储在不同的内存里面哦。

（2）、垃圾回收

垃圾回收(Garbage Collection)：简称GC，Flash Player当消耗的系统内存增多，需要申请内存前，其自动从内存中移除无用对象的过程称为垃圾回收。千万记得，一定是申请内存资源的时候触发，扔掉一个内存资源的时候是不会触发的哦。另外，这个垃圾回收的具体时机我们也是不能控制的。

回到本节开头的问题：什么样的内存才是垃圾内存？

对于垃圾内存的定义，不同垃圾回收策略有不同的定义方式，但都可以有一个比较抽象的概括，那就是不可达的内存叫做垃圾内存。具体一点的话就得先介绍垃圾回收的策略。

（3）垃圾回收策略

Flash Player使用的是和Java类似的垃圾回收策略，一共有两种方式：引用计数法和标记清除法

引用计数法（Reference Counting）

引用计数法是通过计算指向某个对象的引用的数量来确定是否清除该对象。如果一个对象的引用数量为0，表示程序无法访问到该对象，则清除该对象；如果引用计数不为0，则不清除，这种方法运行代价较小。

?
a = new ObjectA();
b = new ObjectB();
b.prop = a;
Reference counts:
    ObjectA:2
    ObjectB:1

如上图，上面的3行代码执行后，对于new出来的这个ObjectA对象来说，先是被变量a引用，然后又被b.prop引用，所以最终的引用次数是2，而new出来的ObjectB只被b引用，所以引用个数是1。

如果这个时候将代码改为：

?
a = new ObjectA();
b = new ObjectB();
b.prop = a;
b = null;
 
Reference counts:
    ObjectA:2
    ObjectB:0

new出来的ObjectB对象的引用次数为0，在执行完上述4行代码后，它被标记为垃圾，稍后的某个时刻就有可能会被回收。

修改一下上面的例子，如下：

?
a = new ObjectA();
b = new ObjectB();
b.prop = a;
a.prop = b;
a = null;
b = null;
 
Reference counts:
    ObjectA:1
    ObjectB:1

虽然我们将a、b都赋值为null，但是由于他们之间的相互引用，导致ObjectA和ObjectB的引用次数还是为1。从引用计数法的角度去看，这两个对象都不属于垃圾资源哦。但从代码角度来说，这两个对象已经没法通过代码去访问操作了，按理说应该属于垃圾了吧。这里出现了问题。问题的关键是引用计数法不能帮我们将所有的垃圾内存都选择出来，那就得另请高明了！

标记清除法（Mark Sweeping）

标记清除法是从程序的根对象（GCRoot）开始，遍历每个引用指向的对象。遍历经过的对象，则将其标记。最后清除所有没有打上标记的对象。这种方法比较彻底，但是遍历所有的对象，想想都知道这个方法效率很低。

打个比方，农田里面种了几棵西瓜，我们顺藤摸瓜，从每棵瓜藤的根开始，将所有连在瓜藤上的西瓜都标记起来，那没有标记的西瓜不就是烂掉的垃圾西瓜，可以扔掉了。

标记清除法确实能解决引用计数法无法解决的循环引用问题，它把关注点放在根对象上，即瓜藤，只要不能通过我们根对象摸到的对象，一律都是垃圾。

如下图，ObjectA和ObjectB纠缠在一起，通过GCRoot摸不到他们，他们就被认为是垃圾，可以在未来回收。

在Flash中，有几个比较重要的瓜藤（GCRoot）如下：

  • 舞台stage
  • 程序主SWF的文档类
  • 静态变量

看了标记清除法后，是不是恍然大悟，我们代码中有很多地方都没有将变量设置为null，也有些地方只有addEventListener而没有removeEventListener，但这些资源依然被回收了，这都是标记计数法的功劳啊。想想，要是每个对象都要在自己即将销毁前将所有引用设置为null，并且将所有用到addEventListener的地方都removeEventListener，那得多累啊。

这样看来，标记清除法非常的厉害，管你什么垃圾，一网打尽，似乎是全能的，就用他了，不用什么引用计数法了，一点儿也不准。但事实上，Flash是两种都有用，这是一种折中的办法，虽然引用计数法不能将所有的垃圾都揪出来，但是他却能在短时间内判断出大部分垃圾内存，而标记清除法虽然无所不能，但是代价却比较高。所以，先用引用计数法快速回收大部分垃圾内存，然后，将标记清除法分段，每次执行一部分，偶尔执行一次。这样既可以保证找到所有的垃圾，又能将CPU消耗降到最低。

现在让我们来小结一下，以上两种判断垃圾的方法都有一个共同的判断依据——引用。引用计数法中引用次数为0的是垃圾，标记计数法中只有被GCRoot直接或间接引用的对象才不会被当做垃圾。那有没有一种情况，就是对象A确实被对象B引用了，但是在回收A对象的时候，我不想因为B对象的引用造成A不能被回收。打个比方，女孩子A有一个男朋友B，A不能回归单身再去寻找其他男生，但是有一种情况可以，就是A和B的情侣关系是比较脆弱的，想分手时就分手。这种关系在Flash中叫弱引用。

（4）弱引用

简单来说，无论是引用计数法或是标记清除法，在判断引用的时候，如果这种引用关系是弱引用，那可以当做没有引用来判断。

回到上面的分手话题，如果男孩B对女孩A的引用是弱引用，女孩A想回归自由身的时候可以毫不顾忌男孩B。但这个时候如果女孩A还有个男朋友C，C对A的引用是常规引用，或者说是强引用，那女孩A是万万不能被单身大军回收的。

在Flash，用到弱引用的地方就只有两个（还好，这种不稳定的关系不常见）： 

  •  addEventListener的第5个属性：名为userWeakReference，设置为true，监听事件将不会影响对象回收。
  •  Dictionary的构造函数参数：名为weakKeys，设置为true，当键为复杂对象时，即使Dictionary存在，键依然可以被回收。注意，这里说的是键，不是值，值是不享受弱引用待遇的。   

其实在我们游戏中，弱引用还是很少用到的，毕竟这种不稳定关系的编程模型很少遇到。所以大家还是要记得，持有别人的引用一定要记得清除，否则别人一辈子都不能回归自由身的。特别是某些男生持有一堆女孩子的引用，不放开，还不断申请新的女孩子。导致其他男生没有资源，结果会导致天下大乱的。

Flash有强大的内存回收机制，这就让我们的Flash程序员不用像C++程序员那样整天关注内存的分配与释放，大大减轻了程序员的负担。但是，这并不意味着，Flash完全解决了内存泄露的问题，相反，如果对Flash的垃圾回收机制了解不深，将会导致大量的内存泄露在你的Flash程序中出现。

（5）内存泄露

Flash中的内存泄露是指由于疏忽或是错误导致已经不再使用的内存不能被垃圾回收器判断成垃圾内存，造成系统内存浪费的情况。它是内存管理的最后一个内容，也是真正的重点，前面花了那么多笔墨介绍内存的分配与回收就是为了避免我们的程序出现内存泄露。毕竟很多新人甚至部分老程序员都经常犯这个错误。

要想避免内存泄露，我们就得让不再使用的垃圾内存真正变成垃圾。由上面的内容可以得知，让垃圾内存的引用次数减为0，或者让垃圾内存参与的引用环和GCRoot不存在交集，形成孤岛，这都是解决问题的方法。看起来很简单，但往往知易行难，一不小心，内存泄露就出现在你的程序中了。

这里我总结了几个容易让新人的程序出现内存泄露的地方： 

  •  没有移除事件监听器 ：有用到addEventListener的地方一定要记得removeEventListener。事件监听器是一个比较隐式的引用，这个地方出现内存泄露的情况是最多的。

    ?
    public class A
    {
        public function A()
        {
            //当A不再使用时，对象B是static的
            B.instance.addEventListener(BEvent.MY_EVENT, handleEvent);
        }
     
        private function handleEvent(evt:BEvent):void{
            //to do
        }
    }
     
    public class B extends EventDispatcher
    {
        private static var _instance:B;
        public function B()
        {
        }
     
        public static function get instance():B{
            return _instance ||= new B();
        }
    }
     
    public class BEvent extends Event
    {
        public static const MY_EVENT:String = "MY_EVENT";
        public function BEvent(type:String, bubbles:Boolean=false, cancelable:Boolean=false)
        {
            super(type, bubbles, cancelable);
        }
    }

    如上面的代码，虽然B没有明显的地方引用了A，但是继承了EventDispatcher的B实质上是有一个监听者列表，所有监听B的事件的对象都被B引用着。所以只要B存活，监听者就不能被回收。解决方案就是在A即将销毁的时候调用一次removeEventListener。

    ?
    public class A
    {
        public function A()
        {
            //当A不再使用时，B依然使用
            B.instance.addEventListener(BEvent.MY_EVENT, handleEvent);
        }
         
        private function handleEvent(evt:BEvent):void{
            //to do
        }
         
        public function dispose():void{
            B.instance.removeEventListener(BEvent.MY_EVENT, handleEvent);
        }
    }

    当然记得你的dispose()一定要掉哦，很多同学记得写removeEventListener，却非常遗憾的忘记调用了。

  • 子对象引用父对象，而父对象没有释放：常见的是显示列表中，某个显示元件里面监听stage的事件。

    ?
    import flash.display.Sprite;
    import flash.events.KeyboardEvent;
    public class A extends Sprite
    {
        public function A()
        {
            stage.addEventListener(KeyboardEvent.KEY_DOWN, onKeyDown);
        }
         
        private function onKeyDown(evt:KeyboardEvent):void{
            //to do
        }
    }

    如上图，A的函数onKeyDown被stage引用，导致A不能被释放。解决方法和事件监听类似，添加dispose函数，同样，记得调用哦：

    ?
    public function dispose():void{
        stage.removeEventListener(KeyboardEvent.KEY_DOWN, onKeyDown);


  • 对象被Array、Object、Dictionary引用：这个问题的出现有几种情况，比如只记得往Array、Object、Dictionary里面塞东西，没有删除。或者将Array当成Object、Dictionary这样的key-value数据结构来使用，最后还是以数组的方式从Array中拿数据等等。
     

 最后，需要提醒的是，内存泄露的出现往往不只是浪费内存，如果有ENTER_FRAME、 TIMER这些事件被垃圾内存监听着，那就会有额外的CPU资源跟着遭殃。下一节将介绍CPU资源的管理。

二、CPU

游戏卡最明显的特征就是帧率下降，帧率是一秒钟刷新的总帧数，简称FPS。我们游戏的帧率是25，客户端游戏一般是60。如果按照25帧1秒来计算，一帧就有40ms的时间去处理代码执行和图形图像的渲染。帧率下降的原因就是40ms不够用，需要额外的时间，比如Flash执行完一帧的代码和渲染足足花了100ms，那我们帧率就会下降的10。这是完全不能接收的，用户可以明显的感觉到游戏不流畅。

说到底就是CPU的工作效率是固定的，而要保证25的帧率，那就得降低一帧里面的CPU工作量。在一帧里面，CPU的工作量主要有两个部分：一个是代码的执行，比如ENTER_FRAME、 TIMER事件的处理；另外一个就是图形图像的渲染。基于此，我们将降低CPU工作量的事情划分成降低每帧执行的代码压力和优化每帧图形图像的渲染压力。这里，我们先来看看如何降低图形图像的渲染压力。

1、降低渲染压力

在Flash中显示对象就只有图形和图像，图形就是矢量图，图像就是位图，对于这两个内容需要分开介绍。

矢量图

Flash中的矢量图记录的是点、线和填充。这些数据都是以坐标和公式来存储的，比如Flash矢量图中的曲线就是用二次贝塞尔曲线（Quadratic Bezier）实现的。

这些点、线和填充数据要在显示器上显示，就得换算成具体的像素点。假如舞台是960*560，那就得有537600个点需要计算，假如这些矢量图之间还有各种遮挡、半透明、放缩、滤镜等复杂的关系和设定，那计算这些像素数据给显示器将是非常消耗CPU资源的一件事情。那矢量图是不是没有优点了呢，当然不是，它有很多的优点，比如：存储量小、缩放不影响质量、做动画方便等等。特别是我们网页游戏，大量的数据需要下载，合理使用矢量图，将会大大提高游戏的下载体验。

既然没法避免使用矢量图，计算就一定有，那减少计算量将是合理使用矢量图的关键所在。哪些东西会增加矢量图的计算呢？下面将一一道来。

  • 复杂图形：一颗500*400的大树，每片叶子纹理都很清楚，最后缩小成50*40，变成了一个图标，很多数据都看不清楚。这就是典型的浪费渲染资源。
  • 半透明：这个东西非常消耗CPU资源，特别是大量图形重叠的时候。
  • 渐变：这种有过渡计算的设定都是非常消耗CPU的。
  • 混合模式：将原有的颜色填充数据做处理后再显示，额外的CPU消耗。
  • 滤镜：效果越好，越吃资源，这个设定还会消耗额外的内存资源。
  • 多重嵌套：造成显示列表的树状结构庞大，元件间关系复杂，遍历都会消耗大量CPU。
  • 补间：每帧都有平滑过渡的改变，增大CPU消耗。
  • 等等

以上内容只是优化矢量图方法里面的一部分，现在不懂也没事，以后会慢慢明白的。更多的技巧需要大量的经验积累，新同学在未来的实践中多用心，就会发现了。

有时候矢量图怎么优化都还是造成CPU的大量消耗，甚至游戏卡到不能玩了。怎么解决？

有没有人想过既然矢量图需要大量计算才能显示，那能不能直接保存计算完成后的数据，到时候直接显示不就解决问题了么。想到这点的同学真是聪明，这种存储结构的确存在，他就是位图。

位图

位图记录的是真正能显示的像素点，显示时没有额外的CPU消耗，直接显示即可。

位图的优点是，消耗CPU低，但是缺点也十分明显，存储量大，一般是矢量图的10倍以上。这种缺点在网页游戏中会被体现的特别明显，存储位图的文件会非常大，直接导致游戏过程中下载的等待时间很长。所以位图一般在不得已的情况下才使用，比如人物模型，这种东西每帧都在变化，而且当同屏幕人数增多时，会有大量的数据需要显示，如果用矢量图实现将会消耗大量的CPU。

看来位图是一定要用的，有哪些需要注意的呢？

  • 缩放：位图缩放会有锯齿，影响视觉效果，并且也会消耗不小的CPU资源，慎用。
  • 旋转：同样有锯齿，有CPU消耗，慎用
  • 平滑：对位图进行平滑处理会有大量差值计算，CPU消耗非常大，慎用。
  • 透明：和矢量图一样的原理，重叠时消耗大量CPU，慎用。
  • 导出类：对位图设置导出类，会降低CPU消耗。原因是普通的位图放在舞台上，实质是用一个shape以位图填充的方式显示，有性能消耗。有导出类后，继承了Bitmap，这个位图才会调用专门针对位图的渲染器了，这种渲染方式就比较高效了，可以用。

位图需要注意一下，既然选择了位图这种消耗用户等待时间来换取CPU的方式，就要将位图的好处发挥到极致，不要在位图上添加缩放、旋转、平滑这些又会降低性能的设置，这是得不偿失的。在未来的工作中，一定要注意。

介绍完矢量图和位图的原理和对CPU影响后，我们需要知道的是，那些地方需要使用位图，那些地方需要使用矢量图，这样才能合理的利用CPU。

合理利用CPU只要遵循一个原则就行，这个原则就是：所以场景中的东西都应该用位图，所有模式面板（模式面板打开后就不能操作角色行走做动作）都可以采用矢量图。理由很简单，场景中元件的渲染是不可控的，它会随着同屏幕人数的增多而消耗更多的CPU。面板就不同，打开面板时，将面板后面的显示对象全部停止，或者移出显示区域，这样渲染消耗的CPU资源就只剩下面板了。基于这个原则，游戏中使用位图的地方有：人物衣服、宠物、怪物、场景物件、掉落物品等等。

其实，降低渲染压力还有几个方法：

  • 减少显示列表元件数量：在渲染过程中需要遍历显示列表，他是一个树状结构，当然是层次结构越简单，同层数量越少，遍历的时候CPU压力越小哦。所以在写程序的过程中一定要尽量保证显示列表的干净、整洁。
    冻结和解冻对象：冻结和解冻对所有对象都很重要，但对显示对象尤其重要，在冻结的时候stop，解冻的时候play。即使显示对象不再位于显示列表中并正在等待作为垃圾回收，其代码仍然占用大量 CPU。使用 REMOVED_FROM_STAGE 和 ADDED_TO_STAGE 事件冻结和解冻对象。

    如下例子中，对于继承了MovieClip的类A，监听从舞台移除的事件REMOVED_FROM_STAGE和加载到舞台的事件ADDED_TO_STAGE，移除时stop，加进舞台时play。

    ?
    public class A extends MovieClip
    {
        public function A()
        {
            addEventListener(Event.REMOVED_FROM_STAGE, onRemoveFromStage);
            addEventListener(Event.ADDED_TO_STAGE, onAddToStage);
        }
         
        private function onAddToStage(evt:Event):void{
            play();
        }
         
        private function onRemoveFromStage(evt:Event):void{
            stop();
        }
    }


降低渲染压力就介绍到这里，下面介绍怎样减少代码造成的CPU压力。

2、降低代码执行压力

在一帧的执行周期里面，有一部分时间是留给我们代码的。一般来说，游戏必然会通过监听ENTER_FRAME或TIMER来实现游戏中物体的移动和变化，这就是我们常说的心跳机制。也就是说心跳一下，马上去更新游戏中物体的位置、动作、状态等等。心跳是固定时间发生的事情，大部分情况下，我们是采用ENTER_FRAME事件来触发心跳，游戏的大部分代码都是从这个事件开始执行，好的游戏，只会有一个地方监听了ENTER_FRAME事件，这样便于管理和实现诸如暂停、慢放等功能。

我们在这个事件的处理函数里面加上如下代码：

?
var started:Number = getTimer();
......
心跳触发的各种处理代码
......
var runTime = getTimer() - started;

以上代码的runTime就是我们代码消耗的时间了，一目了然，建议刚学Flash游戏的同学，一定要在你的心跳处理函数里面加上这段代码，时刻检测自己程序的执行时间。以帧率25为例，单帧执行时间不应该大于40ms，而大部分时间留给了渲染，代码的执行时间最好控制在1ms以内，特殊情况下可以控制在5ms以内。超过这个数值，就应该检查自己的代码，是什么地方出现了问题。

说了检测方法，那我们编程过程中有什么地方可以优化的呢，下面将列举一部分出来：

  • Flash Player 10 引入了 Vector 类, 其读写访问速度比 Array 类更快
  • 利用mouseEnabled 和 mouseChildren 属性禁用鼠标交互
  • 对于TextField ,使用 appendText() 方法, 而不要使用 += 运算符
  • 将重复使用的变量定义在循环之外
  • 使用位移运算符（<<,>>）替换乘除运算符（＊，/）
  • 尽量停掉所有不在显示屏幕中的物件的动画和代码，这部分代码如：怪物AI、屏幕外的人、NPC的状态等等，这种技巧在大屏幕游戏里面最常见，龙斗士里面就有采用，有兴趣的同学可以来咨询
  • ......

以上只是代码优化的冰山一角，很多技巧需要在实践中慢慢积累并灵活运用。

三、文件

文件在我们网页游戏里面可谓是举足轻重，对文件资源的控制其实就是对下载时间的控制，下载时间长是非常影响游戏体验的，所以文件资源管理的重点就是控制文件大小。在我们游戏中，文件资源主要是swf、xml和音乐文件。对这些文件资源的管理，有两个点需要注意：

  • 文件大小的控制
  • 文件里面的代码和数据的安全

1、文件大小的控制

网页游戏不同于客户端游戏，它运行所需要的代码和资源都是从服务器上下载完成后才能运行的，在还没有下载完成之前，游戏某些功能是不能玩的。特别是第一次接触我们游戏的用户，他需要下载相对比较完整的游戏资源和代码才能进入游戏。这个时间是比较长的，如果用户等得太久，直接放弃了，纵使游戏多优秀也是浮云，人家根本看不到。所以对游戏文件大小的控制是非常重要的。

先将游戏的文件资源分成三类：swf、xml和音乐文件

swf文件

用Flash CS打包的swf文件，一般情况下是压缩保存的，Flash只支持ZLIB压缩算法，效率还不错。这种文件打包了美术资源和代码，一般来说，代码所占的文件大小跟动不动几百K的美术资源相比，简直弱爆了，所以代码大小的控制，先放着，重点放在资源文件的大小上面。

对于swf文件大小需要注意的地方，我又总结了几点，如下：

  • 资源文件中装的主要是矢量图、位图及动画数据。当然还有一些对游戏来说是垃圾的数据，比如便于搜索引擎识别的Metadata、线上环境的调试数据，这些通通可以砍掉，只需要在Flash CS里面设置一下即可

  • 位图资源，swf文件中最大的资源就是位图资源，动不动就上百上千K，它是swf文件大小优化的重点。一般情况下，我们可以采用压缩位图的方式进行优化，设置压缩比率，找一个视觉上刚好过得去的压缩比率。另外，有时候，美术那边经常是将一张大位图放在fla文件里，最后只用了这张位图的一个角落，这是极大的浪费，可以使用位图工具裁剪后再编译。

  • 矢量图资源，swf中有大量的矢量图资源，优化起来有两个原则：1、将复杂简单化；2、删除不在可视区域内的资源。

xml文件

xml文件一般用来存储数据，游戏里面用的比较多，比如龙斗士里面就有上几十个xml文件。这些文件如果不压缩，将会很大。比如龙斗士里面的装备数据，不压缩，足足有4M+，太大了，所以压缩是必然的。

一般压缩有两种方式：

  • 以embed方式压缩进swf文件；
  • 直接对xml文件进行压缩，下载后调用as的ByteArray.uncompress()函数解压缩。

无论是哪一种压缩方式，算法都是一样的，必须是ZLIB，没办法，Flash就支持这个，其他压缩算法他不理。不过还好，4M的xml数据采用ZLIB压缩后编程300多K，还是挺不错的。

音乐文件

尽量使用mp3文件，容量最小，这个大家应该都知道。值得注意的，mp3要打包到swf文件里面，不能直接使用，因为mp3这种格式如果商用是要收费的，放到swf文件里面就不用了，Adobe帮我们付钱了。

2、加密

swf文件格式是公开的，里面的资源、代码都可以通过Sothink、ASV等工具直接反编译，一览无余。这样很容易暴露我们财富，最危险的是，让别人知道我们的消息格式，外挂就会满天飞了。所以对游戏文件加密是及其重要的。

加密可以分为三种：1、加密文件的字节序列。2、混淆代码。3、让反编译工具失效

加密文件的字节序列有很多种方法，不固定。简单的如每个字节加上一个固定的或是有规律的数值，不打乱原始字节的位置，复杂的除了将原始字节加密后，还打乱了字节的顺序。这样有心人即使拿到你的游戏文件也没法反编译了。但是，往往道高一尺，魔高一丈。而就目前来看，破解游戏的阵营有很大的优势。所以加密字节序列似乎不是特别可靠，这里就得说一说另外一种加密方式——混淆代码。

混淆代码的宗旨是让反编译者不能轻松的阅读我们代码。具体做法很多，比如在代码中添加与业务无关的代码，或者将代码中的类名、变量名、函数名等字符串用另外的字符串代替，这些做法都是为了加大反编译者阅读代码的难度，让其看不懂代码。

最后一种方法就是让反编译工具失效，一般是让反编译工具在反编译过程中因为无法识别某些字节而崩溃，这种方法只能针对特定的反编译工具，比较局限，但效果非常好，可以阻碍破解者拿到我们的美术资源和代码资源。

破解别人加密文件是一件非常有意思的事情，特别是破解后观摩别人的代码，这是一种很好的学习方式。毕竟游戏编程百家争鸣，我们只是其中一家，了解的东西很有限，只有集百家之所长才能让自己立于不败之地。

      String的特殊之处在于，AS中对String的初始化有3种：

      1、var str1:String = "just test";

      2、var str2:String = new String("just test");

      3、var str3:String = String("just test");

      变量str1、str2、str3都在栈中，这个无异议。

      对于第1种方式，在栈中有一个字符串池子，将"just test"存放起来，然后str1指向的是栈中的这个地址。如果有var str4:String = "just test";这样的语句时，先在栈中创建变量str4，然后到池子中寻找这个变量是否存在，如果存在，则将str4指向这个地址；没有的话，就在池子中加入新来的的字符串，然后新的地址赋给这个str4。这样处理可能是为了节省内存吧。

                  string a = "just test";

                  string b = "just test";

                  Console.WriteLine(ReferenceEquals(a, b)); //输出true

                  int c = 1;

                  int d = 1;

                  Console.WriteLine(ReferenceEquals(c, d)); //输出false

      可以看出a和b指向的是同一个对象了，当然不排除as和c#的虚拟机机制不一样。
