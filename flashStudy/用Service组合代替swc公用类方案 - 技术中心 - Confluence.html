用Service组合代替swc公用类方案

转至元数据结尾
  • 由 未知用户 (qiusheng)创建, 最后修改于十二月 20, 2012
转至元数据起始

题目好难命……

描述：

宠物面板，要显示一只宠物的血量攻击防御天赋等信息。在宠物背包要显示，在小屋要显示，在孵化成功时要显示，在……要显示……显示的内容是一样的，但样式背景子面板可能一样可能不一样。

历史演化：

开始时这个面板只出现在小屋，所以小屋中有个面板的导出类PetAllInfoPanel。后来背包也要一样的面板。小屋和背包是两个文件，也为了减少异步操作带来的复杂性，面板就拷了过去。但导出类代码没有必要两份。就出现了两个文件编了同一份导出类代码的情况。

出现问题：

小屋的这个面板做了点小改动，改了代码。两个文件有同包名同类名的类时，Flash Player会默认使用先加载那个。我们每周发版只编译变更文件，小屋编译了，背包没有编译，背包里面板代码还是旧代码。运行时先打开小屋，再打开背包，代码就是新的，背包里面板没改，代码找不到资源，报错；反过来，代码是旧的，小屋里新代码无法生效。

知道这个问题时，很容易解决，背包也修改面板并编译。但知道这个问题前呢？作为新接手小屋的开发，还有测试，都不知道背包和小屋有如此强的耦合。作为测试，看到的现象是打开背包的面板有时会报错，有时不会，根本不知道是因为进了小屋引起的。光是重现BUG都要花很长时间。作为开发，也不知道修改小屋里的一个类，还会影响到背包的功能。试想这种情况泛滥，修改一个类，潜在可能要编译多个文件，修改代码如履薄冰，人心惶惶。

想偷懒的解决方法：

最简单方法，还是把代码再拷一份到背包，这样两边独立，修改互不影响。但每次修改bug，就要修改两份。有人会觉得那就修改两份吧，也不会很难。这种想法是非常危险的。代码是会扩展的，臭味会随之扩展。当多个地方都要这个面板时，代码就不是两份而是多份了。更可怕的是，当你修改一份代码的bug时，还不知道有多少份相同或相似的代码要修改。想想都觉得很臭！

有人会考虑把代码移到公共的包，例如命名为mmo.pet.common的包。看到这么“大”的包名，可以联想到它可能被多个文件编译进去，修改者就会变小心点，修改前查一下引用。靠人为去保障的事情，都是无法保障的！何况Flash里直接将导出类设为这个类时，它根本不在文本文件里，连引用和全文搜索都查不到。

让那些错误的取巧方法随风而去吧。

正统方法：

了解swc强大之处的人，也许已经想到了。将这个类放到有swc的文件里，这样这个类只会被编译一次。场景基类SceneBase就是一个典型例子。但宠物面板，应该放在哪个swc里呢？宠物算是奥拉星核心玩法，放核心swc不为过。那其他像周刊基类这些，似乎就不妥了。swc不断扩大，就会产生互相引用等问题。而且swc过大，可能导致其他文件编译失败（貌似与编译的类数量和编译机器内存有关）。为了避免swc无止境膨胀，尽量使它简单，我们才有了IOC，提供给各服务间互相调用。详细见Flash进阶系列——IOC及其应用

既然IOC可以提供服务，那能不能解决这个问题呢？

IOC里取到的都是类的实例，不能像SceneBase直接用来继承。问题就变成了，用什么代替继承？

重点来了：

没错，前面都是废话。

面向对象设计原则：多用组合，少用继承。

细想这句话，仿佛就是在说组合是可以在某些情况下代替继承的。

看看简化了的原来的PetAllInfoPanel

PetAllInfoPanel
?
public class PetAllInfoPanel extends Sprite
{
    public function PetAllInfoPanel()
    {
        petAttrIconContainer = this.getChildByName("petTypeIconContainer") as MovieClip;
        petIconContainer = this.getChildByName("iconcontainer") as MovieClip;
    }
  
    public function setObj(o:Object)
    {
        TextField(this.t0).text = o.txt;
    }
}

外部它的父面板对它的调用

外部调用
?
private var petAllInfoPanel:PetAllInfoPanel;
  
function
{
    petAllInfoPanel = this.infopanel;
  
    petAllInfoPanel.setObj(evt.params.pm);
}

下面改成用组合，把面板作为面板功能服务的其中一个属性：

PetAllInfoPanelController
?
public class PetAllInfoPanelController extends EventDispatcher implements IPetAllInfoPanelController
{
    private var panel:Sprite;
 
    public function PetAllInfoPanelController()
    {  
    }
  
    public function init(panel:Sprite):void
    {
        petAttrIconContainer = this.panel.getChildByName("petTypeIconContainer") as MovieClip;
        petIconContainer = this.panel.getChildByName("iconcontainer") as MovieClip;
    }
 
    public function setObj(o:Object)
    {
        TextField(this.panel.t0).text = o.txt;
    }
}

做成Service，我们无法控制构造，不能在构造传入参数，所以把构造方法的事情放到init方法里。将面板在Init方法作为参数传入，原来所有this与面板的相关的操作改成this.panel，如this.getChildByName改成this.panel.getChildByName。（为什么wiki代码不能改颜色……）

再看外面的调用：

外部调用
?
 private var petAllInfoPanel:IPetAllInfoPanelController;
 
 
function
{
    petAllInfoPanel = ServiceContainer.getService(IPetAllInfoPanelController) as IPetAllInfoPanelController;
    petAllInfoPanel.init(this.infopanel);
 
    petAllInfoPanel.setObj(evt.params.pm);
}

对比原来，只有类型由原来的实现类改成了接口，构造时将实际面板传入。其他调用方法的地方都没变。就像给面板穿了一层衣服，原来面板的功能，现在都在衣服上了。钢铁侠啊！

这样修改，调用的地方只用改构造的地方，修改量少，又不用swc，完美解决前面所有问题！
