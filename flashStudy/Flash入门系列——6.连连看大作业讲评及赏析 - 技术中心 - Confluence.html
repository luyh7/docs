Flash入门系列——6.连连看大作业讲评及赏析

转至元数据结尾

转至元数据起始

如果你看这篇文时，还没有交大作业，请先看完Coding板块如何写代码系列，运用到你的代码里，让改作业的人更好理解你的代码。

然后看完常见问题集锦 - Coding、常见问题集锦 - 前端，在自己的代码里不要犯同类错误。

最后，如果你真的没做完大作业，务必在完成大作业之后再看本文。没经过思考就得到的知识是不牢固的。没写过差的代码，是不懂欣赏好代码的，如同没吃过豪客来，就不知道太古仓洛奇先生吧的苹果木牛排有多好吃！

虽然说，无论你看或不看，代码就在那里。但是，请注意节操。

 

－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－比起代码一点都不华丽的分隔线－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－

自从布置大作业以来，每届新人经常犯的错误，主要集中在编码和设计的问题上。

编码上的问题，在常见问题集锦 - Coding，养成习惯就好。如果你发现自己的问题没有记录，也欢迎补充，为后人提个醒。

本文更多集中在方法和设计的问题上。

很多人戏称我们为码农。我认为，码农做的是给定一个方法输入输出，然后仅负责实现这段编码，或者是长期做机械式重复编码，例如装电脑、人工做运维配置、复制代码等。我们不是码农，我们贯穿软件工程中从需求分析一直到测试之前的流程，某些工作甚至要考虑运维因素。我们是软件开发工程师，不是码农。按Ctrl＋C，Ctrl＋V时应该感到愧疚。刚才复制常见问题集锦的链接过来，我双手在颤抖。

见过最恶劣的设计，或者称不上有设计，是整个游戏只有一个类。典型就是事先没想清楚，想到什么就做什么的。

流程：

软件开发还是应该遵从流程。需求分析，概要设计，详细设计，这三步都是不能跳过的。写设计文档会消耗很多时间，而且我们的需求改动非常大（互联网公司大体如此），文档的更新甚至会跟不上代码更新。因此我们提倡无文档，当今很火的敏捷开发也提倡无文档。无文档不意味着跳过这些步骤，还是需要在大脑里思考，画画草图、类图，只是最后的结果不需要记录正式文档。

需求分析：将非程序员提出的需求转换成程序员的思维，俗称建模。我们公司在这个阶段需要做的更多。我们的用户是儿童，要理解儿童真正的需求。笼统来说就是好玩、有趣、炫的东西，具体来说包括勇气、成功、美丽、被认同等等。需求文档通常是项目组的策划同事提出，他们也许也是新人，也许本身不太会玩游戏（笔者长期完虐策划组各种小游戏），也许不够时间想得很周全。拿到需求文档时，我们要理解这份需求的目的是什么，是否能达到目的，是否能受用户喜欢。在此基础上，我们可以和策划商量，让这个功能更丰富，更好玩。Do the Right Things 比 Do the Things Right 更重要。

连连看作业让大家发挥创意的目的也在此。只是实现点两个图标能连就消失的功能很简单，是否有什么遗漏？例如洗牌，没有这个功能，遇到无法连时根本玩不下去，原来的需求是不完整的（出现这种功能点遗漏后果很严重，策划和开发都有责任。当然这是我布置作业的坑，我没有责任）。还有，我们能否做更多？例如关卡，图标消失后按指定方向移动填满空位，提示功能，无法连时自动洗牌功能等等。

概要设计：想想需要哪些包、哪些类，每个类职责大概是什么。复杂的系统画画类图，复杂的交互画画时序图。有经验后，在这步大概能评估开发时间和风险了。

详细设计：针对重点、难点想想能否实现。看需求一定要仔细，曾经也试过一个细节需求颠覆整个设计的。连连看，重点难点应该是判断能否连的算法。当一个细节无法实现时，就意味着这份需求无法完成，这时怎么办？写需求的人不懂程序，有些做法我们很难实现甚至无法实现，但是我们可能可以换一种做法轻松的做出类似的效果。举个不太恰当的例子：策划可能知道给用户发消息是要写数据库的（有些计算机专业的去做策划会以为自己很懂开发），需求要求给12月21日后登录的每个用户插入一条消息。几千万用户每人插一条数据库记录，会给数据库带来压力，其实，只要判断一下用户最后登录时间是在12月21日前，就给他发消息就行了，完全不需要插入数据库的。

不知不觉说了策划很多坏话。合作久了就能体会，总会觉得有些又难做感觉效果又不好的功能点。但策划写一份需求确实不容易（不信你自己写写看），总是否定会打击他们士气。有个观点要遵循：当你觉得一个功能点很糟糕，请提出更好的；否则，请闭嘴（这句式很像翻译过来的，但确实是我总结的）

我的做法：

需求分析阶段，大概想想要些什么功能。每个人想法不一样，这部分不多说。

概要设计阶段，游戏总有一个主面板，定义一个GamePanel类，协调各子面板面板；图标有很多种类，用Icon类管理；IconMap面板是最主要的子面板，用来存放图标，响应用户点击操作；连接的算法会比较麻烦，单独抽出Algorithm类；LevelManage类管理关卡间切换和初始化，LevelConfig配置关卡信息；MovieManager管理一些动画效果。

详细设计阶段：连接算法，很多人直接的想法是：直接一条支线能否连接，经过一次拐点能否连接，经过两次拐点能否连接。然而，我是一个有风骨的程序员。我第一个想法就是广度优先搜索：第一层搜出四个方向上所有能连的点，非空白点判断是否目标点，空白点则进入下一层，垂直改变方向后继续搜索这两个方向上的点；重复三层后，就能找到两次拐点内能连的所有点。这样做的好处是什么？假如做个奇葩点的连连看，要求可以三次拐点甚至四次连接的，只要增加搜索层数就可以了。当然，这有点过度设计。最大的好处还是在于，这更贴近程序员的思维。

中途的问题和解决：

问题一：位置记录

图标保存在一个二维数组里，第一维表示行，第二维表示列。鼠标点击图标时，要知道图标所在的行与列，将这个操作做成一个方法，返回值就需要有两个，是做不到的。这时就要用到面向对象的第一个特征，封装。再想想要知道图标所在行和列这个操作，要遍历二维数组查找整个图标实例所在，效率低下。将坐标记录在图标对象上，就可以直接获取了。正好，可以将坐标封装在图标类里。

Icon
?
public class Icon extends MovieClip
{
    public var type:int;
    public var posI:int;
    public var posJ:int;
     
    public function Icon(type:int)
    {
        this.type = type;
        this.stop();
        var cl:Class = ApplicationDomain.currentDomain.getDefinition("linklinklook.view.PetIcon") as Class;
        var mc:MovieClip = new cl() as MovieClip;
        mc.gotoAndStop(type + 1);
        this.addChild(mc);
    }
}

注：标准来说，这里应该做成private，然后用get和set方法，完整的封装。错误示范，切勿模范。仅在赶时间且影响范围较小时使用。

很多人会用一个二维数组记录图案类型type，再用一个二维数组记录图标的显示对象，或者直接将显示对象存在一个显示容器里。将这两个也封装，构造时传入type生成显示对象，何乐而不为？

 

问题二：重复代码

二维数组表示行和列，但在屏幕坐标系里，x轴是从左到右，y轴是从上到下。而且，x、y坐标与图标的大小和间隔有关。这种对应关系的代码，写第二次都会头痛，必须果断抽象出方法。

这个方法应该放哪里呢？给图标设置行列时，就要设置它在屏幕上的位置。仅与Icon类相关的事情，与Icon就是紧耦合的，应该封装在Icon类里。

还是Icon

public static var ICON_WIDTH:int = 50;
public static var ICON_HEIGHT:int = 50;
  
public static function setXYfromIJ(d:DisplayObject, i:int, j:int, middle:Boolean):void
{
    d.x = ICON_WIDTH * j;
    d.y = ICON_HEIGHT * i;
    if (middle)
    {
        d.x += ICON_WIDTH / 2;
        d.y += ICON_HEIGHT / 2;
    }
}
  
public function setPos(i:int, j:int):void
{
    this.posI = i;
    this.posJ = j;
    setXYfromIJ(this, i, j, false);
}
setXYfromIJ是一个静态方法，因为爆炸效果、提示动画都要加在与图标相同的位置，可以公用。middle是因为注册点位置可能是中间也可能是左上角，需要调整。

当行列发生改变时，坐标必须同时发生改变，setPos将这系列操作封装起来。直接操作posI,posJ将破坏封装导致行列与坐标不一致，这就是前面将posI,posJ做成public的弊端。

问题三：类重量级

做到算法时，先验证点击的两个图标是否相同图案，然后交给广度优先搜索判断这两个位置能否连接。搜索算法里只关注位置，与图案无关，更与显示对象无关。搜索算法里每个节点都用到Icon这么重量级的类，会非常奢侈。而且即使是没有图标的空位置，还是要搜索。这时候，就应该将位置单独抽象出来了。

本来抽象出Pos类，Icon类是应该重构成持有Pos类，不再持有PosI和PosJ两个基本类型的。重构不完全的遗物，作为错误示范使用，切勿模仿。

性能调优：

位图：

刚做出来的游戏，播放消除动画和连线动画时非常卡。

想起一件重要的事，背景是矢量图。虽然它本身不动，动画在它上面播放，导致要一起重绘。果断缓存为位图。

将不动的或只有位移的背景改成位图或缓存成位图，是提升性能最快捷又最有效的方式。必做！

缓存池：

游戏每玩一段时间，就会突然卡一下。这种一般是垃圾回收造成，垃圾回收运行时严重消耗资源，导致屏幕跳帧。这时需要减少对象的生成和销毁。

用Flex profiling跟踪，查看哪些资源是经常被生成销毁的。

在用工具前，我先大胆猜测：动画是非常消耗内存的，爆炸和连线动画每次播放都生成，播放完后销毁。

用工具后，事实证明了我的猜测是正确的，也是残缺的。在搜索过程中大量创建Pos类，其内存消耗与动画相当。

直觉很重要，但不要依赖。当有工具能监控时，用工具证明你的直觉。

创建一个池（Array），创建对象时，看看池里有没有，有就取出来，没有就再创建；用完后，清空状态，放回池里。要记得清空状态，否则下次取出的对象数据就有问题了。

缓存池
?
public class Pos
    {
        private static var posPool:Array = [];
 
 
        public static function createPos(i:int, j:int):Pos
        {
            if (posPool.length > 0)
            {
                var p:Pos = posPool.pop();
                p.setPos(i, j);
                return p;
            }
            return new Pos(i, j);
        }
         
        public function dispose():void
        {
            posPool.push(this);
        }
    }

缓存池有几种形式，例如是事先创建好对象在池里，还是用时再创建；池有没有固定大小。按情况使用。

亮点赏析：

单一职责：

来看下游戏的文档类。很多人把主要逻辑都放在文档类，比如说连接算法、动画效果、倒计时等。

单一职责原则：一个类只有一个引起它变化的原因。

文档类本身继承显示对象，只有显示上的变化，才能导致这个类修改。连接算法完全与显示无关，是绝对不应该放这里的。即使是显示上的功能，也应该划分得更细。一个类承担了太多责任，会增加这个类的复杂性，对调试、测试、阅读都带来不便。比如说，修改一个关卡的代码，却可能导致连接算法的bug，这种设计是不合理的。

再看我的文档类DocClass，只有一句代码。我们的项目由多个swf组成，使用时才下载，下载后文档类会无法释放，所以要求项目内所有文件文档类不能拥有美术资源，代码尽量简单。

GamePanel是游戏的主面板，只负责处理用户交互的流程框架，而具体实现都放在了其他类。

很多小游戏要用到EnterFrame事件控制（刚好这个可以不用）。全部代码写在一起，会破坏单一职责原则；每个类单独监听EnterFrame事件，做“暂停”功能就会有很多重复代码用来判断是否暂停。推荐用统一入口，每个类分别实现的方式：

统一入口，分别实现
?
this.addEventListenner(Event.EnterFrame, onEnterFrame);
  
function onEnterFrame(evt:Event):void
{
    if (this.isPause)
    {
        return;
    }
    this.player.onMove();
    this.bullet.onEnterFrame();
    this.background.onEnterFrame();
}

另外隆重推荐mmo.common.event.EnterFrameManager。奥拉星原创，方便事件统一暂停和移除。但奥拉星小游戏都是copy了很多份代码来用，这是错误的（小游戏制作流程原因，不解释），只需用create方法创建一个实例就可以了。

多用组合少用继承：

继承是“is a”的关系，组合是“has a”的关系。组合拥有更高的灵活性，也更有利于类的细分。

B类想使用A类的部分属性和方法，又有自己的方法。如果用继承，就违反了Liskov原则（子类必能能替换掉它的基类）。其实想用A类的“部分”属性和方法时，这里已经是A类过大的bed smell了。拆分部分B需要的功能到A1类，其他放在A2类，A由A1,A2组合，B组合A1，问题轻松解决。

还有，AS和Java都是不支持多继承的，用组合也能解决。奥拉星人物模型Player类，它既是一个可以移动的类Movable，也是可以穿衣服的类Dressily，但又不能多继承。经过几次变动（历史原因，不解释），目前是实现IMovalbe和IDressily接口，继承Movable，将Dressily作为属性组合。

回到连连看，IconMap用来处理图片矩阵，没有继承显示对象，而是以组合形式将container作为属性。除了组合的好处，这里也将显示对象和逻辑拆分开来。pureMVC框架里View层也是这种做法（千万别用pureMVC框架，学学思想就好）。

来看下Flash导出类和基类造成的血案（有点深，看不懂可以略过，若干时间后再来看）：
很久很久以前，游戏周刊需要做个点击物件找齐五个得奖励的小功能，理所当然，代码写在导出类里。后来每期周刊都要这个功能，就小功能抽象出一个基类，每期周刊导出类继承这个基类。从其他语言来看，这是非常好的重构。但我们每期周刊都是单独的swf文件，基类被编译进每个文件。Flash加载多次同包名同类名的类时，会保存第一份，后面的都失效。有一天，新周刊有新功能，并且以后每期都要，于是又加进基类里。但旧的周刊，我们是不会重新编译的（编译整个奥拉星所有前端文件需要整整一天，所以我们只编修改过的）。用户先打开旧周刊，再打开新周刊，基类就是旧的，就没有了新功能。

组合如何解决？将原来的基类作为管理类，构造时周刊文档类作为参数传入，再将管理类作为service编入公用文件。使用时，取出管理类service，传入周刊导出类。除了构造方法，外部使用这个管理类和原来使用基类的接口是完全一致的。

但是后来功能上取消了往期周刊，就没有这些问题了。

关于这个问题，可以看用Service组合代替swc公用类方案

开发封闭原则：

游戏第二关，图标消除后向下靠拢，第三关向左靠拢.。这个功能怎么实现？

图标移动功能
?
private function removeIcon(pos0, pos1):void
{
    ...//removeIcon
    switch(level)
    {
        case 1:break;
        case 2:moveDown(pos0,pos1);break;
        case 3:moveLeft(pos0,pos1);break;
    }
    ...
}

当要增加第四关向中间靠拢的功能时，就必须新增moveMiddle方法，并且修改switch里的代码。这就破坏了原来的格局，用设计原则术语就是对修改开放。

开放封闭原则：软件的实体应该是可扩展，不可修改的。对扩展开放，对修改封闭。

要做到这个原则，就要依赖抽象，面向接口编程而不是面向实现。

不同的关卡，只是移动的策略不同，就把移动的策略抽象出接口。

IMoveStrategy
?
package linklinklook.strategy
{
    import linklinklook.view.IconMap;
     
    public interface IMoveStrategy
    {
        function move(iconMap:IconMap, removeI0:int, removeJ0:int, removeI1:int, removeJ1:int):void
    }
}

不同的移动策略NoneStrategy，DownStrategy，CrowdStrategy实现这个接口。每关构造IconMap时，传入移动策略实例。

IconMap
?
public function IconMap(height:int, width:int, moveStrategy:IMoveStrategy)
{
    this.moveStrategy = moveStrategy;
    ...
}
  
public function removeIcon(i0:int, j0:int, i1:int, j1:int):void
{
    ...//removeIcon
    this.moveStrategy.move(this, i0, j0, i1, j1);
    ...
}
  

当新增关卡时，新增策略实现这个接口，配置文件构造这个策略就可以了。扩展了新功能，又不用修改IconMap的代码。

看到Strategy，学过设计模式的人可能会想到策略模式。严格来说不算。策略模式更强调用组合的形式创建丰富的对象，想想鸭子的例子，抽象出飞和叫两个接口，通过各种飞的实例和叫的实例组合出不同的对象。学习设计模式，不应该太在意设计模式。它只是对使用面向对象基本原则解决问题的方法的总结。只要牢记面向对象设计原则并时刻运用，你就是无招胜有招的设计大师。

DRY

Don't repeat yourself。最容易理解但又很难完全做到。即使只有一行相同甚至有点不同的代码，都可能是重复代码。实践中慢慢体会吧

 

结束语

好的代码不只是给机器看的，而是给人看的，是给人容易看懂的。

要写好代码除了学习，更重要的还是实践。

设计的道路很漫长，笔者也只是设计之路上一只迷途小羊，文中有不当地方欢迎指出和修改。
