2017.11.5-2017.11.10


自评

      这周做了什么：

  1. 木蚂蚁的正式接入、调试。
  2. 正式接入蓝叠，正式测试没有问题，一次过。
  3. 靠谱的包又出问题了。。。


      本周的收获：

  1. 原来木蚂蚁和靠谱我都还没有做测试服的支付联调，搞的要重新测试调试，很消耗时间，以后要把这些事件的完成状态记录下来，养成这种习惯，避免再出现忘记是否联调的问题。
  2. 木蚂蚁华为P9闪退是因为木蚂蚁提供的jar包有问题，他们修复之后这个问题解决了。
  3. 木蚂蚁在OPPO上调不出支付页面和退出框，支付页面现在还没解决，退出框已经找到原因了，是因为木蚂蚁的悬浮窗权限太高，导致挡住了退出框。

导师评

由于后端可能会一时对接很多渠道，会导致某个渠道断节，所以对于自己负责的范畴上面可以更多地去push下。

木蚂蚁的闪退问题的处理上面，需要总结一下。例如如何去更好的定位问题，该问题是出现在第三方身上，还是我们自身的问题上。我当时去定位的时候，就是在第三方的demo工程中一步一步地加入我们自己代码，最后发现，即使不加入我们代码，他们依然报错。

对于退出框的问题，纯粹是经验的问题，可以涨一个经验。



2017.10.30-2017.11.3


自评

      这周做了什么：

  1. 木蚂蚁、靠谱的正式接入。乐视倒闭了，不用接了。
  2. 接了蓝叠，还没接完，等志成搭好测试环境就可以测试了。

      本周的收获：

  1. 木蚂蚁和靠谱是一个月之前接的，和测试服做过了联调，现在准备正式上线，需要在正式环境测试。来这里实习了三个月，现在看之前对接的代码还是发现了有一些问题，比如有些渠道给的参数在sdkmanager里面写死了，这样是不行的，这些值应该写在AndroidManifest或者数据库里面。除此之外也有出现代码逻辑上的缺陷，这是我对安卓应用生命周期不了解造成的，这些问题我都找出来并解决了。
  2. 游戏+渠道的正式测试和sdk Demo的测试不太一样，前者更加复杂，有些问题只在特定的机型会出现。下面是这周我遇到的，或者是立芬在测试的时候反馈过来的问题：
    1. 乐视和靠谱无法拉取服务器列表。乐视是因为这个渠道不需要接了，服务器没有搭起来。靠谱应该是因为platformid或者别的参数出了问题，靠谱目前算是有2个渠道，一个是我一个月之前接的版本，一个是新的“发行”渠道，对接的区别只有kaopu_game_config.json和appid不一样，其他诸如代码逻辑、appkey、secretkey都是一毛一样的，为此需要新增一个靠谱（发行）的渠道 ，但是这两个渠道的sdk project是共用的，只需要在打包之前替换kaopu_game_config.json。
    2. 华为P9运行吞天-木蚂蚁在支付的时候，下完单进入第三方支付页面就会立即闪退，其他机型没有出现这种问题，sdk Demo也在P9上有这个问题，报错的log我已经截图下来。至于是什么原因还需要问一下木蚂蚁那边的负责人员。
    3. 在OPPOR7 有支付宝客户端，选择支付宝进行支付，调不起支付界面。这个问题还没处理。
    4. 游戏提示网络连接失败。原因是包名不对，打包系统的游戏渠道信息应该填上相应的包名。
  3. 蓝叠的问题主要集中在后台的签名那里，特别是支付，卡了很久。这个渠道的特殊之处是下单的时候需要在蓝叠验证，验证成功之后才能跳转到支付页面，验证需要的参数除了订单信息之外还有把订单信息拼凑并进行MD5加密得到的签名，下单的时候这些参数从后台传过来，然后再由sdk前端去蓝叠验签。由于baitian.common公用包里的加密处理不完善，处理中文的时候会得出不正确的结果，所以后台得出的MD5签名和蓝叠得出的MD5签名不匹配，造成验签失败。这个问题目前已经解决，就等测试环境测试支付了。
  4. 在安卓测试机上打开Log的方法：修改机器上/Btsdkgame文件夹下对应游戏-渠道包文件夹下的/Debug/debug.txt，加一行debugLog=true，就可以了。

导师评

 1.游戏上线之后，或许会暴露更多的问题，由于android的机型多，特定的机型获取会出现特定的问题，所以一般出现这种问题，应该要学会如何快速的判断问题的根本原因。

        方法1：先用渠道的demo看看是否会复现。

        方法2，3，4.。。。。。。自己总结下。

 2.蓝叠的问题，很多时候需要前端人员去了解后端对接工作，例如后端需要什么参数是前端提供的，讲究一个配合的问题，需要多沟通，包括与第三方平台的人员沟通。

3.安卓开log的方式，有没有想过，有其他更好的方法呢？

2017.8.07-2017.8.11


自评


      这周做了什么：

  1. 做了J04作业第一题。
  2. 根据家涛分配给我的需求，修改打包平台更新渠道版本的功能，以前只能上传经过处理的zip渠道压缩包，现在可以上传渠道apk包。

      本周的收获：

  1. 熟悉了DBManager模块的各种用法，批量操作确实比使用多次单条sql语句效率更高，因为它能够减少连接数据库的次数，节约很多时间。因为这次的数据库作业题目，我在提高大批量数据插入效率那一块卡了很久，对这个问题，我跟数据库课程的讲师吴泽雄老师进行了讨论，这让我对数据库的索引建立过程以及DBManager的批量操作有了更加深刻的理解。另外，我还学会了用explain来分析sql查询语句。
  2. 之前在打包系统更新渠道版本的时候，需要上传新的zip渠道压缩包，而这个zip包的来源是编译打包得到的渠道apk包，经过对apk包的反编译、修改配置文件等步骤再压缩得到zip包的成品。渠道包从apk到zip的这个过程放到了程序中处理，这样就减少了人力成本。所以现在打包平台在打包的时候，会先看看数据库记录（注意，这里不是看渠道包的路径）上有没有相应的zip，如果没有，说明要把apk转成zip，就需要执行上面这些过程。第一次生成了zip包之后，以后打包的时候可以直接拿这个zip包了。

导师评

这周的需求，除了需要掌握打包系统的打包流程之外，更需要修改这个流程用于日后的更新，也能进一步通过apk的编译来理解一个apk工程的细节，例如资源文件，smail代码，manifest配置之类的。

2017.7.31-2017.8.04


自评


      这周做了什么：

  1. 做了J02的两次作业，是关于并发的。
  2. 修改了打包系统的新增渠道版本功能，现在能将新增的渠道版本添加到所有拥有该渠道的游戏上。

      本周的收获：

  1. 做J02并发作业的收获是
    1. 知道了synchronized的用法，synchronized的最小粒度是Object。
    2. 需要注意的是，synchronized锁住的不是Object本身，而是Object指向的一个对象，例如 Object lock = objectList[0]; synchronized(lock){}; 这时候锁住的是objectList[0]，假如此时令lock等于其他Object，那么锁并不会改变，依旧锁住objectList[0]。
    3. 顺便学了一点python爬虫。
  2. 做J04作业的时候没有直接用MySQL语句注入的方式来执行sql语句，而是效仿apkpack项目的做法，对每个表建立DAO来进行这个操作。这样做花的时间比较久，但是能让代码好看点。

导师评

  2. 这周迁移服务器遇到了挺多问题的，但是最后都解决了。
    1. 首先是修改配置文件修改了半天，因为迁移之后tomcat放在了和之前不同的目录下，所以，原来的这些配置的绝对路径都要改变；
    2. 新的Linux服务器的环境变量把JAVA_HOME设成了jre的路径，我把它改成了jdk的路径；
    3. 然后就是除了迁移tomcat服务器之外还要把/home/site下的apkpack迁移过去，因为这是这个web应用前端相关的文件，这是我在看tomcat里的server.xml才发现的；
    4. 用到的8108端口不知道被谁占用了，所以我把tomcat服务器端口改成了8208；
    5. 熟悉了Linux的系统指令之后，从tomcat到web应用，一层一层看配置表，顺藤摸瓜，把缺少的android sdk、工作区文件都拷过来，再加一些环境变量，就迁移成功了。

导师评

代码重构是一个持续的过程，为了该工程不那么痛苦，就需要写一个代码的时候，就需要考虑日后可预见的变化，这些经验慢慢积累吧。

迁服这个事情，那些文件要修改和那些环境变量要配置，这个需要总结一个文件出来：如打包环境迁移所需操作。希望能把所做的事情更有章节地说出来。
